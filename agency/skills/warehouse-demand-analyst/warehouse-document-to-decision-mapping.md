# Warehouse Document-to-Decision Mapping

## Зачем нужен этот skill

Этот skill заставляет кладовщика работать не от файлов как таковых,
а от цепочки:

`документ -> извлечённые поля -> вычисленные метрики -> управленческий вывод`.

Если эта цепочка не materialized, агент начинает путать:
- источники данных;
- техническое чтение;
- управленческий смысл.

Цель skill:
- сделать понятной роль каждого входящего документа;
- закрепить, какие именно цифры из него извлекаются;
- определить, что именно должен увидеть руководитель в финальном отчёте.

## Общий discipline gate

Применять `agency/skills/common/codex-karpathy-discipline.md`.

Для document-to-decision это значит:
- не подменять решение списком файлов;
- не придумывать метрики, которых нет в источнике;
- не расширять manager output техническим шумом;
- проверять, что каждая поднятая цифра ведёт к действию или риску.

## Главный принцип

Нельзя считать документ обработанным, если агент только:
- прочитал файл;
- создал карточку;
- перечислил названия колонок;
- выписал несколько строк.

Документ считается обработанным только когда:
1. понятно, какие поля из него извлечены;
2. понятно, какие метрики из этих полей посчитаны;
3. понятно, какой конкретный управленческий блок из этого появился.

## Обязательный маршрут по каждому документу

### Шаг 1 — Определить тип документа

Агент обязан явно отнести вход к одному из доменных типов:
- `ABC-анализ`
- `Остатки`
- `Продажи`
- `Выручка`
- `Каталог`
- `Накладные PDF`
- `Комментарии / notes`

Если тип неясен:
- документ не попадает сразу в executive-layer;
- агент фиксирует ambiguity в `Проверить вручную`.

### Шаг 2 — Извлечь фактические поля

Агент обязан назвать не “что примерно там внутри”, а конкретные поля.

Пример формата:
- `SKU`
- `qty`
- `revenue`
- `cost_unit`
- `cost_total`
- `abc_class`
- `supplier_name`
- `invoice_date`

Если поле ожидалось, но не извлеклось:
- это считается factual gap;
- gap должен быть отражён в card/report.

### Шаг 3 — Посчитать derived metrics

Из extracted fields агент обязан построить метрики.

Примеры:
- `critical stock`
- `reorder qty`
- `top movers`
- `weak movers`
- `price delta`
- `cost share`
- `gross margin`
- `margin pressure`
- `supplier risk`

Нельзя перескакивать от сырых полей сразу к совету без явно понятной метрики.

### Шаг 4 — Превратить это в manager output

Каждая метрика обязана попасть в один из понятных блоков:
- `Срочно заказать`
- `Планово заказать`
- `Не заказывать / сократить`
- `Проверить вручную`
- `Изменение цен`
- `ABC и себестоимость`
- `Продажи и ассортимент`

Если метрика не доходит до manager output,
агент обязан честно понимать, что документ прочитан только частично.

## Карта документов

### `ABC-анализ`

Извлекаем:
- `SKU`
- `abc_class`
- `qty`
- `revenue`
- `share`
- `cost_unit`
- `cost_total`
- `cost_share`

Считаем:
- лидеры `A`
- слабые `C`
- тяжёлые по себестоимости позиции
- позиции с прижатой валовой маржой
- кандидаты на сокращение закупки

Даём руководителю:
- `Лидеры ABC`
- `Тяжёлые по себестоимости позиции`
- `Маржа под давлением`
- `Позиции C-категории под пересмотр`

### `Остатки`

Извлекаем:
- `SKU`
- `qty_now`
- `location`
- `item_group`

Считаем:
- `critical stock`
- `planned reorder`
- `reorder qty`
- `stop-sale risk`

Даём руководителю:
- `Срочно заказать`
- `Планово заказать`
- `Готовые заявки поставщикам`

### `Продажи`

Извлекаем:
- `SKU`
- `qty_sold`
- `revenue`
- `period`
- `location`

Считаем:
- `top movers`
- `weak movers`
- позиционную оборачиваемость
- слабые SKU для сокращения

Даём руководителю:
- `Топ продаж`
- `Слабые позиции`
- часть блока `Не заказывать / сократить`

### `Выручка`

Извлекаем:
- `period`
- `location`
- `revenue`

Считаем:
- различия по точкам
- динамику выручки по периоду
- поддерживающий контекст для продаж и ассортимента

Даём руководителю:
- supporting layer для блока `Продажи и ассортимент`
- factual context для решения по ассортименту

### `Каталог`

Извлекаем:
- `SKU`
- `sale_price`
- `group`
- `cost/margin`, если присутствует

Считаем:
- `price mismatch`
- потенциальную базовую маржу
- необходимость пересмотра цены

Даём руководителю:
- `где плохая цена`
- `что нужно пересчитать`
- supporting layer для `себестоимости/маржи`

### `Накладные PDF`

Извлекаем:
- `supplier_name`
- `invoice_date`
- `invoice_no`
- `SKU`
- `qty`
- `unit_price`
- `amount`
- `confidence`

Считаем:
- `price delta`
- `last purchase price`
- supplier-specific price history
- закупочную базу для cost cards

Даём руководителю:
- `Подорожали`
- `Подешевели`
- `Проверить вручную` для аномалий и low-confidence строк

### `Комментарии / notes`

Извлекаем:
- человеческие пометки
- долги/корректировки
- контекст по нестандартным ситуациям

Считаем:
- нет числовой метрики по умолчанию;
- только factual context / manual review signal

Даём руководителю:
- только в `Проверить вручную`
- или как factual note в supporting layer

## Truthful правило

Если агент не может честно ответить на 3 вопроса:
- `что я извлёк из документа?`
- `что я из этого посчитал?`
- `что увидел руководитель в отчёте?`

значит документный цикл ещё не доведён до идеала.

## Expected outcome

После применения этого skill кладовщик должен уметь:
- объяснить путь любого документа до manager-report;
- быстро находить, где именно ломается цепочка;
- не путать ingestion defect с отсутствием данных;
- не смешивать техническую карточку документа и управленческое решение.
