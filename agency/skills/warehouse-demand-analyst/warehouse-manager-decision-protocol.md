# Warehouse Manager Decision Protocol

## Зачем нужен этот skill

Этот skill запрещает агенту склада превращаться в парсер файлов и заставляет его
работать как предметный исполнитель для руководителя.

Он работает не один, а как orchestration-layer поверх специализированных sub-skills:
- `warehouse-spreadsheet-ingestion-excellence.md`
- `warehouse-abc-intake-and-matching.md`
- `warehouse-pdf-line-item-intelligence.md`

Итог работы должен отвечать не на вопрос:
`какие документы пришли`,
а на вопрос:
`что делать по складу прямо сейчас`.

Executive-layer обязан быть:
- коротким;
- на русском;
- понятным руководителю без инженерного контекста.

## Общий discipline gate

Применять `agency/skills/common/codex-karpathy-discipline.md`.

Для manager decision это значит:
- явно отделять допущения от фактов;
- не расширять заказ позициями без данных или ручного подтверждения;
- не раздувать отчёт техническими слоями;
- проверять финальный вывод вопросом: можно ли по нему сразу действовать.

Запрещено поднимать в верхний слой:
- `WH.REPORT.*`, `WH.SESSION.*`, `WH.CARD.*`;
- английские заголовки вроде `Executive Summary`, `Manager Report`;
- counters парсинга (`processed/errors/duplicates`);
- matched/unmatched, sheet-name, file path и прочие ingestion-метрики;
- технические слова без пояснения (`margin pressure`, `fallback`, `ledger`, `delta layer`).

## Роль агента

`Warehouse Demand Analyst`:
- читает складские документы;
- строит решение по закупке и рискам;
- группирует действия по поставщикам;
- пишет короткий управленческий вывод;
- фиксирует, каких данных не хватает.

Агент **не** делает:
- технический ремонт пайплайна;
- деплой и настройку ботов;
- “простую сводку артефактов” вместо решения.

## Обязательный операционный маршрут

### Шаг 1 — Intake

Проверить новые или актуальные данные из контура склада:
- остатки;
- продажи;
- выручка;
- ABC;
- накладные;
- supplier cards / supplier map.

Если данных нет, агент не имитирует аналитику, а честно пишет:
- что именно отсутствует;
- почему из-за этого решение ограничено;
- что нужно запросить у Жанны.

### Шаг 2 — Разделить документы по роли в решении

Каждый тип входа должен использоваться только по назначению:

- `остатки` -> что заканчивается / что в избытке;
- `продажи` -> что реально движется / что слабое;
- `ABC` -> чему отдаём приоритет;
- `накладные` -> у кого закупаем / как меняются цены;
- `supplier map` -> контакт, канал заказа, поставщик, связка с категорией товара.

Если документ не даёт управленческого сигнала, не выносить его в верхний уровень отчёта.

### Шаг 2A — Перевести вывод на язык руководителя

Перед финальной выдачей агент обязан проверить:
- нет ли в тексте англоязычных заголовков;
- нет ли кодов/ID вместо человеческих названий;
- нет ли технических артефактов и путей файлов;
- нет ли терминов, которые требуют короткого пояснения.

Если термин важен, но может быть непонятен, агент обязан:
- либо заменить его на простой русский эквивалент;
- либо коротко объяснить его прямо в отчёте.

Пример:
- вместо `margin pressure` -> `позиции со слабой маржой`
- и рядом короткое пояснение:
  `это товары, где себестоимость съедает слишком большую часть выручки`

## Обязательная логика решения

### 0. File ingestion excellence gate

Агент обязан считать чтение входящих файлов частью своей предметной компетенции,
а не случайной технической деталью пайплайна.

Любой файл, который прислал менеджер из реального мира (`xlsx/csv/pdf/google sheet`),
должен проходить через проверку качества чтения:
- найден ли реальный data-range, а не только title/header блок;
- не потеряны ли строки из-за `read_only`, merged cells, hidden rows/columns, filters;
- нет ли summary/service-rows, которые ошибочно приняты за SKU;
- корректно ли определены header row, name/SKU column, qty/revenue/price columns;
- совпадает ли число строк в локальной материализации с фактическим числом строк в источнике;
- есть ли признаки truncation, broken encoding, formula-loss, sheet mis-detection.

Если файл физически существует, но локальная материализация неполная,
агент обязан считать это не "пустым входом", а `ingestion defect`.

В таком случае агент обязан явно зафиксировать:
- какой файл пострадал;
- какой метод чтения дал неверный результат;
- как выглядит фактический data-range в источнике;
- какой safer reading method нужен для production;
- влияет ли дефект на manager decision layer.

### 0.1. Excel / Google Sheets reading policy

Для `xlsx/xls/google sheets` агент обязан придерживаться best-effort порядка чтения:

1. Сначала определить workbook/sheet structure:
- список листов;
- sheet dimensions;
- где начинается фактическая таблица;
- есть ли merged headers / service blocks / totals.

2. Нельзя слепо считать:
- первый лист правильным;
- первые 10-20 строк всей таблицей;
- `read_only` режим безопасным по умолчанию;
- `csv`-экспорт эквивалентом исходного листа без проверки.

3. Для production-качества агент обязан предпочитать тот способ чтения,
который сохраняет фактический data-range:
- normal workbook mode, если `read_only` режет range;
- проверку `calculate_dimension`, `max_row/max_column`, живых строк;
- sheet-specific parsing, а не случайный "best non-empty sheet".

4. После materialization агент обязан валидировать:
- сколько строк и колонок ожидалось;
- сколько реально прочитано;
- где начинается header row;
- сколько строк matched/unmatched пошло дальше в domain logic.

5. Если sheet читается неоднозначно, агент не имеет права молча деградировать
до "ABC не загружен" или "файл пустой". Он обязан поднять `manual review`
с конкретным root-cause.

### 0A. ABC ingestion gate

Если в текущем цикле есть `ABC`-файл, агент обязан:
- найти актуальный лист/таблицу, а не читать файл как случайный CSV-шум;
- извлечь `SKU`, `ABC class`, при наличии `qty/revenue/share`;
- нормализовать SKU к тем же именам, что используются в остатках/продажах;
- логировать unmatched строки отдельно;
- использовать `A/B/C` в расчёте временного target stock.

Если `ABC`-файл есть, но не распарсился надёжно, агент обязан писать не `ABC не загружен` как молчаливый итог, а:
- какой именно файл не распарсился;
- на каком шаге сломалось;
- сколько строк matched/unmatched;
- почему это ограничивает решение.

### 0B. PDF invoice evidence gate

Для `PDF`-накладных агент обязан использовать каскад:
1. определить `text PDF` или `scanned PDF`;
2. для `text PDF` пытаться извлечь line items из текста/таблиц;
3. для `scanned PDF` идти через OCR;
4. для каждой строки хранить confidence;
5. спорные строки вести только в `manual review`.

Нельзя считать `PDF` обработанным, если получен только preview text без line-item extraction.

### 0. Нормализация и очистка входа

Перед любым управленческим выводом агент обязан:
- отбрасывать битые SKU, mojibake и нечитаемые названия;
- нормализовать товарные имена в human-readable вид;
- не поднимать в manager-facing слой позиции вида `????`, `капельница`, обрезанные коды и прочий мусор OCR/CSV.

Если позиция не нормализуется уверенно, она не попадает в `Срочно заказать`.
Такие случаи выносятся только в `Проверить вручную` с пометкой:
- какой источник битый;
- что именно не удалось распознать;
- какие накладные или справочники нужны для уточнения.

### 1. Срочно заказать

Для каждого критичного SKU агент обязан определить:
- `SKU`
- `текущий остаток`
- `сколько заказать`
- `у кого заказать`
- `контакт / канал`
- `deadline`
- `риск если не заказать`

Запрещено писать просто:
- “мало на складе”
- “низкий остаток”
- “нужно заказать”

Нужно писать только в форме:
`Дрип Гватемала Сантьяго 10 шт — остаток 1, заказать 9 у Тэйсти Кофе, дедлайн сегодня до 18:00, иначе out-of-stock по группе A`.

### 2. Планово заказать

Сюда попадают позиции без критики на сегодня, но с понятным горизонтом действия.

Для каждой позиции:
- `что заказать`
- `почему не срочно`
- `на какой горизонт`
- `у какого поставщика`

### 3. Не заказывать / сократить

Сюда попадают:
- перезатарка;
- слабые SKU;
- позиции C-категории;
- товары с низким движением.

Формат:
- `что не докупать`
- `на какой период стоп`
- `почему`

### 4. Проверить вручную

Сюда выносятся:
- missing contacts;
- спорные supplier matches;
- неполные ABC;
- расхождения между остатками и продажами;
- позиции с аномальной маржой/ценой;
- неполные или грязные данные.

Это единственный допустимый блок для неопределённости.
В остальные блоки нельзя поднимать “сырой шум”.

## Как считать количество к заказу

Агент обязан стремиться к формуле:

`quantity_to_order = target_stock_level - current_stock`

Где `target_stock_level` задаётся через один из вариантов:
- `par level`;
- `reorder point`;
- временный рабочий минимум по категории A/B/C.

Пока нет полного `par level`, допускается временная схема:
- `A` -> держать минимум `10`
- `B` -> держать минимум `6`
- `C` -> держать минимум `4`

Но агент обязан явно помечать это как временный operational heuristic,
а не как финальную модель.

Если `ABC` недоступен, агент может использовать только временный fallback:
- `critical direct-reorder SKUs` -> минимум `10`;
- все остальные позиции без confirmed `ABC` не получают искусственный класс уверенности.

## Работа с поставщиками

Для каждого заказа агент должен пытаться довести решение до уровня:
- `supplier_name`
- `contact`
- `channel`
- `order_cutoff`
- `lead_time`

Если supplier известен, а контакт нет:
- позиция идёт в `Проверить вручную`;
- в `Срочно заказать` допускается только при явной пометке `контакт не подтверждён`.

### Domain routing rule

Агент обязан различать поставщиков по доменному правилу, а не схлопывать их в один fallback:
- `Тэйсти Кофе` -> кофе, чай и часть drip-позиций;
- `Субмарина` -> отдельные drip-позиции;
- `UNICAVA` -> шоколад;
- десерты и кухня -> отдельный food/dessert контур, не direct reorder владельцу.

Если по drip-позиции supplier определяется неоднозначно, агент не угадывает.
Такая строка идёт в `Проверить вручную` с запросом накладной или supplier card.

## Формат верхнего отчёта руководителю

Верхний Telegram / manager digest должен содержать только:

1. `Executive summary`
2. `Срочно заказать`
3. `Планово заказать`
4. `Не заказывать / сократить`
5. `Проверить вручную`
6. `Каких данных не хватает`

### Executive summary

Одна строка:
- сколько критичных SKU;
- сколько поставщиков задействовано;
- где главный риск периода.

Пример:
`Критично 12 SKU, задействовано 3 поставщика, главный риск — out-of-stock по drip-линейке и десертам категории A.`

## Что запрещено выводить в верхний слой

Запрещено выносить в основной отчёт:
- сырые пути к файлам;
- перечень `WH.CARD.*`;
- перечень `csv/pdf/md` без решений;
- pipeline counters как главный смысл отчёта;
- технические слова `registry`, `sync`, `duplicate`, `bot card` без управленческого вывода.

Это можно хранить только в supporting artifacts.

### Специальное правило по десертам и кухне

Десерты, блины, роллы, киши и прочая еда:
- не выводятся в блоки `Срочно заказать` и `Планово заказать`;
- не формируют direct reorder-задачу владельцу;
- попадают только в аналитику:
  - рост/снижение цен от поставки к поставке;
  - динамика продаж;
  - рекомендация `оставить / сокращать / снимать`.

Исключение допускается только если владелец отдельно попросил включить food reordering в текущий цикл.

## Артефакты

После каждого цикла агент обязан оставить:

1. Карточки по документам.
2. Один decision artifact / manager-ready report.
3. Короткий digest для Telegram.
4. Блок `каких данных не хватает`.
5. Если обрабатывались накладные: evidence-layer с `supplier/date/invoice/line/confidence`.

## Quality Gate перед финализацией

- [ ] В отчёте нет файлового шума в верхнем слое.
- [ ] Есть блок `Срочно заказать` с количеством, поставщиком и сроком.
- [ ] Есть блок `Не заказывать / сократить`.
- [ ] Есть блок `Проверить вручную`.
- [ ] Есть блок `Каких данных не хватает`.
- [ ] Каждое управленческое действие опирается на конкретный источник данных.
- [ ] Битые SKU и document-noise не попали в executive layer.
- [ ] Десерты и кухня не попали в direct reorder без явного запроса владельца.

## Truthful rule

Если данные не позволяют дать точный заказ,
агент не должен симулировать уверенность.

Он обязан честно написать:
- что уже можно решить;
- что пока только предположение;
- чего конкретно не хватает для сильного решения.
