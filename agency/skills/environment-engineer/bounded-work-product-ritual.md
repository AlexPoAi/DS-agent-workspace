# Skill: Bounded Work Product Ritual

## Зачем нужен

Этот skill нужен, когда агент работает в живом codebase с несколькими репозиториями
и должен не просто "что-то сделать", а вести работу через чистые bounded slices.

Главная цель:

- не оставлять работу размазанной по грязному дереву;
- не перескакивать через открытие/закрытие РП;
- не смешивать чужие хвосты со своим scope;
- доводить slice до отдельного truthful `commit/push`.

## Когда запускать

- когда задача требует новых файлов или правок в репозитории;
- когда есть риск смешать несколько доменных контуров;
- когда дерево уже грязное;
- когда пользователь явно требует идти через ритуал рабочих продуктов.

## Не запускать когда

- нужен ответ без правок файлов;
- нужен микрофикс в одном файле без открытия нового slice и пользователь явно не
  требует ритуал;
- задача уже находится внутри ранее открытого и ещё не закрытого bounded WP.

## Каноническая последовательность

### Шаг 1 — Открой bounded WP

Сначала зафиксируй:

- какой именно `WP` открывается;
- какова цель этого slice;
- что входит в write-set;
- что сознательно не входит.

Нельзя начинать implementation до явного определения slice.

### Шаг 2 — Назначь исполнителя

До правок явно объяви:

- primary agent;
- support agent, если нужен;
- почему выбран именно этот агент.

Если задача повторяющаяся и у текущего агента не хватает capability:

- сначала материализуй или обнови его skill;
- не подменяй его новым агентом только ради скорости.

### Шаг 3 — Оцени дерево и границы

Перед правками проверь:

- какие репозитории уже грязные;
- какие файлы относятся к твоему slice;
- какие хвосты чужие или параллельные.

Принцип:

- работать только в своём bounded scope;
- не трогать чужие изменения;
- не "заодно" чинить соседние хвосты.

### Шаг 4 — Реализуй только свой slice

Во время работы:

- держи изменения в рамках объявленного WP;
- не расширяй scope без нового согласования;
- если обнаружен новый важный слой, открывай следующий WP отдельно.

### Шаг 5 — Truthful closeout

Перед закрытием WP зафиксируй:

- что реально materialized;
- что не сделано;
- какие хвосты остаются;
- что является следующим честным bounded шагом.

Запрещено закрывать WP как `done`, если результат только частичный.

### Шаг 6 — Отдельный commit / push

После truthful closeout:

- коммить только свой slice;
- не захватывай runtime-артефакты и чужие хвосты;
- push делай отдельным шагом после проверки diff.

## Жёсткие запреты

- Нельзя начинать file edits без открытого bounded WP.
- Нельзя смешивать два независимых slice в один commit.
- Нельзя прятать незавершённость за формулировкой "почти готово".
- Нельзя коммитить runtime-артефакты только потому, что они изменились рядом.
- Нельзя трогать чужое грязное дерево без отдельного WP.

## Короткая формула

`Open WP` -> `Assign agent` -> `Bound scope` -> `Implement` -> `Truthful close` -> `Commit/push`

## Шаблон результата

```markdown
# Bounded slice

## WP
- id:
- goal:
- write-set:
- not-in-scope:

## Agents
- primary:
- support:

## Result
- materialized:
- not done:
- next bounded step:

## Git
- commit:
- push:
```
