# Skill: Park Official Document Cycle

## Зачем нужен

Этот skill нужен, чтобы любой официальный документный цикл по Парку доводился до рабочего состояния, а не останавливался на стадии словесного плана или полупустого черновика.

Skill покрывает полный путь:
- определить адресата;
- найти канонические реквизиты;
- собрать clean-document;
- сохранить его в правильное место;
- связать с Pack;
- зафиксировать отправку, входящий и follow-up.

## Общий discipline gate

Перед созданием или правкой официального документа применять:
`agency/skills/common/codex-karpathy-discipline.md`.

Для Park-документов это значит:
- не угадывать адресата, реквизиты и правовой статус;
- не расширять письмо новыми требованиями без явного scope;
- не менять соседние `COMM/DOC/WP` карточки "заодно";
- проверять clean-version и Pack/Drive readback перед статусом `ready-to-send`.

## Когда запускать

- Когда уже есть черновик письма, но его нужно довести до `ready-to-send`
- Когда нужно подготовить новое официальное обращение по Парку
- Когда документ должен быть сохранён в Google Drive и связан с Pack
- Когда нужно вести цикл `draft -> sent -> waiting -> answered`
- Когда нужен контроль входящего номера и срока ответа

## Не запускать когда

- Нужно проектное решение по архитектуре или конструктиву
- Нужно юридическое заключение по существу
- Нужно общее стратегическое решение по проекту недели
- Нужно просто прочитать контекст Парка без документного действия

## Входы

Минимальный входной пакет:
- актуальный `PARK.WP.*`, к которому относится обращение
- черновик письма или описание предмета обращения
- канонический источник реквизитов `Терра`
- `PROJECT-STATUS.md`
- `CONTEXT.md`

Дополнительный пакет:
- прошлые письма по похожей теме
- карточки `COMM` и `DOC`
- входящий номер по связанному процессу, если он уже есть

## Выходы

Skill должен уметь производить:

1. `Clean Draft`
2. `Drive Filing Record`
3. `Pack Registration Update`
4. `Dispatch Record`
5. `Follow-up Record`

## Каноническая последовательность

### Шаг 1 — Уточни контур документа

Определи:
- тема письма;
- адресат;
- желаемый тип результата:
  - содержательный ответ;
  - подтверждение непричастности;
  - переадресация;
  - назначение ответственного;
  - входящий номер.

### Шаг 2 — Подтяни канонические реквизиты

Найди и проверь:
- ФИО отправителя;
- организацию `Терра`, если она используется как отправитель;
- адрес;
- телефон;
- email;
- данные участка и объекта.

Критично:
- если реквизиты не подтверждены, не делать вид, что документ готов.

### Шаг 3 — Собери clean-version

Вставь в документ:
- адресата;
- реквизиты отправителя;
- точную формулировку вопроса;
- перечень приложений;
- дату;
- статус документа.

### Шаг 4 — Сохрани clean-file

Сохрани документ в правильную папку Google Drive или иной канонический storage.

После сохранения зафиксируй:
- название документа;
- место хранения;
- ссылку;
- версию.

### Шаг 5 — Зафиксируй в Pack

Обнови:
- `COMM` или `DOC` карточку;
- реестр, если документ становится значимым артефактом;
- текущий `PARK.WP`, если этот документ является частью исполнительного цикла.

### Шаг 6 — Переведи статус

Используй один из статусов:
- `draft`
- `ready-to-send`
- `sent`
- `waiting`
- `answered`

Не пропускать промежуточные состояния.

### Шаг 7 — Зафиксируй отправку и follow-up

После отправки записать:
- дату отправки;
- канал;
- входящий номер, если получен сразу;
- ожидаемый срок ответа;
- дату follow-up.

## Quality gates

- Нельзя оставлять письмо без проверенных реквизитов отправителя
- Нельзя считать локальный markdown финальным документом без clean-version
- Нельзя сохранять документ в Drive без последующей фиксации ссылки в Pack
- Нельзя отправлять внешний документ без явного согласования пользователя после просмотра clean-version
- Нельзя ставить `sent`, если не зафиксирована дата/канал отправки
- Нельзя оставлять `waiting` без follow-up даты
- Если ответ пришёл, но не по существу, статус не `done`, а новый цикл `follow-up / escalation`

## Связанный support-skill

Если вопрос адресата и предмет уже определены, а документ нужно довести
именно до официального стандарта подачи, запускать дополнительный skill:

- `rf-official-correspondence-polish`

Он отвечает за:
- шапку справа;
- отсутствие пустых реквизитов;
- чистовую приёмку документа под печать и подачу.

## Связь с контуром

- Upstream: `Park Architect`, `Strategist`, пользовательский factual input
- Main consumer: `Park Permitting & Infrastructure Coordinator`
- Downstream: `Strategist`, `PROJECT-STATUS`, follow-up циклы по согласованиям

## Критерий завершения

Skill отработал правильно, если:
- документ действительно доведён до `ready-to-send` или `sent`;
- место хранения документа известно;
- Pack знает про этот документ;
- следующий follow-up не хранится только в памяти.
