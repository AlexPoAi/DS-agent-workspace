# Domain Scope: Какие СЫРЫЕ файлы читает экономист (финансовый аналитик)

## Назначение

Явный реестр файлов которые экономист **ОБЯЗАН** читать и анализировать для построения Finance View.
Это предотвращает дублирование работы с кладовщиком.

## Независимость от кладовщика

**Кладовщик и экономист работают независимо.**
- Экономист НЕ читает результаты кладовщика (WH.SESSION, WH.REPORT)
- Экономист читает СЫРЫЕ файлы (те же что кладовщик, но в своих целях)
- Оба могут работать параллельно

---

## Файлы которые ЧИТАЕТ экономист

### ✅ ОБЯЗАТЕЛЬНО ЧИТАТЬ

#### 1. Продажи_*
**Паттерн:** `Продажи_YYYY-MM-DD_YYYY-MM-DD.xlsx`

**Зачем:**
- Рассчитать выручку по точкам (turgeneva, lugovaya, samokisha)
- Определить топ-продаж и слабые позиции
- Выявить динамику продаж (рост/падение) vs предыдущий период
- Основание для unit-экономики

**Действия:**
1. Читать: Дата | SKU | Количество (шт) | Сумма (₽) | Точка
2. Агрегировать по точкам:
   - `∑ выручка за период по каждой точке`
   - `выручка / дни = средняя выручка в день`
3. Агрегировать по SKU: топ-20 по выручке
4. Сравнить с предыдущим периодом (если есть архив): `(текущий - прошлый) / прошлый * 100%`
5. Выявить аномалии (просадка >20% от среднего)

**Статус:** Обязателен на каждый цикл

---

#### 2. Выручка_*
**Паттерн:** `Выручка_YYYY-MM-DD_YYYY-MM-DD.xlsx`

**Зачем:**
- Подтвердить выручку из файла Продажи (кросс-верификация)
- Определить структуру выручки по точкам (%)
- Выявить кассовые ошибки или расхождения
- Основание для P&L расчета

**Действия:**
1. Читать: Дата | Точка | Сумма (выручка) | Возвраты (опционально)
2. Рассчитать по каждой точке: `выручка за период`
3. Рассчитать долю каждой точки: `выручка_точки / ∑выручка_всех * 100%`
4. Сравнить сумму выручки с данными из файла Продажи:
   - Если расхождение > 1% → отметить как риск, возможна ошибка учета
5. Рассчитать среднюю выручку в день: `∑выручка / дни`

**Статус:** Обязателен на каждый цикл (часто совпадает с Продажи)

---

#### 3. Накладные_*
**Паттерн:** `Накладные_YYYY-MM-DD_YYYY-MM-DD.pdf` (обычно PDF)

**Зачем:**
- Извлечь фактическую себестоимость (COGS) — реальные цены закупки
- Рассчитать фактическую маржу: `маржа = (цена_продажи - фактическая_себестоимость) / цена_продажи`
- Выявить разницу между маржой в каталоге (расчетной) и маржой по фактическим накладным (actual)
- Выявить изменение цен закупки (inflation/deflation сигналы)

**Действия:**
1. Разбить PDF по накладным (OCR + pdfplumber + ручная проверка при low confidence)
2. Для каждой накладной:
   - Определить поставщика (Тэйсти / Субмарина / UNICAVA / другой)
   - Выделить позиции и суммы (quantity × price)
   - Рассчитать `total_cogs_for_supplier`
3. Агрегировать по поставщикам: `COGS_total = ∑COGS_per_supplier`
4. Рассчитать среднюю COGS на рубль выручки: `COGS / выручка`
5. Рассчитать gross_margin: `1 - (COGS / выручка)`

**Статус:** Обязателен на каждый цикл (основание для маржи)

---

#### 4. Каталог_*
**Паттерн:** `Каталог_YYYY-MM-DD.xlsx`

**Зачем:**
- Получить справочные цены закупки и продажи (для маржи расчёта)
- Соотнести SKU с категориями (product mix analysis)
- Выявить смену цен vs предыдущий каталог
- Рассчитать расчётную маржу (как baseline для сравнения с actual)

**Действия:**
1. Читать: Артикул | Название | Цена закупки | Цена продажи | Поставщик | Категория
2. Рассчитать по каждому SKU:
   - `catalog_margin = (цена_продажи - цена_закупки) / цена_продажи * 100%`
3. Агрегировать по категориям:
   - `средняя маржа по категории`
   - `% продаж по категории` (если есть данные)
4. Сравнить каталог с предыдущим периодом:
   - Какие цены изменились
   - На сколько % выросла/упала цена закупки
5. Выявить SKU с отрицательной маржей (убыточные позиции)

**Статус:** Обязателен на каждый цикл (справочник артикулов)

---

#### 5. ABC анализ_*
**Паттерн:** `ABC анализ_YYYY-MM-DD_YYYY-MM-DD.xlsx` или `ABC анализ*.xlsx`

**Зачем:**
- Определить product mix (какой % выручки от A/B/C)
- Выявить концентрацию выручки (80/20 правило)
- Подтвердить данные из файла Продажи (ABC должен быть построен на Продажах)
- Выявить shift в составе топ-позиций vs предыдущий период

**Действия:**
1. Читать: SKU | Класс (A/B/C) | % выручки | Количество проданных | Маржа (если есть)
2. Агрегировать:
   - `% выручки класса A` (должно быть 70-80%)
   - `% выручки класса B` (должно быть 15-20%)
   - `% выручки класса C` (должно быть 5-10%)
3. Выявить какие SKU переместились между классами vs прошлый период (движение)
4. Если маржа есть в ABC:
   - Выявить класс A с низкой маржой (проблема: много продаж, слабая маржа)
   - Выявить класс C с высокой маржой (возможность: мало продаж, хорошая маржа)

**Статус:** Обязателен на каждый цикл (основание для product mix analysis)

---

### ❌ НЕ ЧИТАТЬ (это читает кладовщик)

| Файл | Почему не читать | Кто читает |
|------|------------------|-----------|
| Остатки по складам_* | Это управление складом, не финансы | Warehouse Demand Analyst |
| Отчет по инвентаризации_* | Это сверка физических остатков, не финансы | Warehouse Demand Analyst |

**ВАЖНО:** Экономист НЕ смотрит результаты кладовщика (WH.SESSION, WH.REPORT). Это независимые потоки.

## Если файла нет?

**Статус Finance View:** `partial`

| Файл отсутствует | Действие | Как отметить в Finance View |
|-----------------|----------|--------------------------|
| Продажи | Невозможна выручка по точкам | Статус partial, "Продажи не поступили" |
| Выручка | Используем Продажи, но это не 100% точно | Статус partial, "Выручка не поступила, использованы данные Продаж" |
| Накладные | Используем расчётную маржу из каталога | Статус partial, "Накладные не поступили, маржа расчётная (каталог)" |
| Каталог | Невозможна маржа расчёта | Статус partial, "Каталог не поступил" |
| ABC анализ | Используем Продажи для топ-SKU анализа | Статус partial, "ABC не поступил, использованы данные Продаж" |

**Пример в Finance View:**
```markdown
## Статус
partial

Отсутствующие данные:
- Накладные (Дата: не поступили) → маржа расчётная (каталог), не фактическая
- ABC анализ отсутствует → product mix определён из Продаж

Риск: фактическая маржа может отличаться от расчётной на ±5-15%
```

## Нормализация (ЕДИНАЯ для обоих агентов)

### Point slugs (обязательно одинаковые)

Из `drive-folders-vkoffee.yaml`:
```yaml
point_slugs:
  тургенева: turgeneva
  карьерный: lugovaya
  самокиша: samokisha
```

**ВСЕГДА использовать эти slug'и во всех расчетах и таблицах.**

### Margin types (ОБА типа маржи, но чётко разделены)

| Тип маржи | Источник | Используется где | Название в выводах |
|-----------|---------|-----------------|-------------------|
| Расчётная маржа | Каталог (цена_продажи - цена_закупки) | Справка, baseline для сравнения | `catalog_margin` или `теоретическая маржа` |
| Фактическая маржа | Накладные (реальная себестоимость) | Finance View, P&L, unit-economics | `actual_margin` или `реальная маржа` |

**Пример в Finance View:**
```markdown
## Маржа по точкам
| Точка | Выручка | Маржа (расчётная) | Маржа (фактическая) | Дельта |
|-------|---------|-----------------|-------------------|--------|
| turgeneva | 1.2M ₽ | 72% (каталог) | 68% (по накладным) | -4 п.п. |
```

## Проверка перед завершением цикла

Перед тем как выдать Finance View:

- [ ] Продажи прочитаны → выручка по точкам рассчитана
- [ ] Выручка прочитана → данные выручки подтверждены (кросс-проверка)
- [ ] Накладные прочитаны → COGS рассчитан → фактическая маржа рассчитана
- [ ] Каталог прочитан → артикулы идентифицированы → расчётная маржа рассчитана
- [ ] ABC прочитан → product mix проанализирован
- [ ] Finance View создан с полной структурой (по точкам, по сети, аномалии)
- [ ] Статус: done или partial (с объяснением почему)
- [ ] Finance View передан в financial-consultant для verdict

## Синхронизация с кладовщиком

**Кладовщик и экономист работают НЕЗАВИСИМО:**
- Кладовщик читает: Остатки, Инвентаризация, ABC, Каталог, Накладные, Продажи
- Экономист читает: Продажи, Выручка, Накладные, Каталог, ABC

**Файлы которые читают оба:** Продажи, Выручка, Накладные, ABC, Каталог
- Это OK! Каждый извлекает свои срезы:
  - Кладовщик: SKU, остатки, поставщики, приоритеты закупки
  - Экономист: Выручка по точкам, маржа, аномалии продаж

**🚨 КРИТИЧНО: КТО ПЕРЕМЕЩАЕТ ФАЙЛЫ**

- **Кладовщик:** ❌ НЕ трогает папки Drive
  - Читает из "Новые документы"
  - Обрабатывает
  - Пишет результаты в PACK-warehouse
  - Оставляет файлы в "Новые документы" (НЕ перемещает)

- **Экономист:** ✅ ТОЛЬКО ОН перемещает файлы в "Обработано"
  - Читает из "Новые документы" (кладовщик их там оставил)
  - Обрабатывает
  - Пишет результаты в DS-finance-private/business-finance/views/
  - **ПОСЛЕ завершения обработки: перемещает ВСЕ файлы**
    - На Drive: "Новые документы" → "Обработано"
    - Это делает ВРУЧНУЮ через скрипт или Google Drive API
    - Перемещаются ВСЕ файлы (даже те что сам не читал):
      - Остатки, Инвентаризация (читал кладовщик)
      - Продажи, Выручка, Накладные, ABC, Каталог (читали оба)

**Почему так:**
1. Один "источник истины" для перемещения (экономист) → нет гонки
2. Файлы доступны для обоих агентов до конца
3. Экономист работает ПОСЛЕ кладовщика (логически):
   - Кладовщик пришел, прочитал свое
   - Потом экономист пришел, прочитал свое
   - Потом экономист архивирует (конец цикла)
4. Четкий момент завершения: когда экономист переместил файлы → цикл done
