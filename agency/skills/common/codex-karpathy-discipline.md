# Skill: Codex Karpathy Discipline

## Зачем нужен

Этот общий skill снижает четыре типовые ошибки LLM-агентов:
- молчаливые допущения;
- избыточные абстракции;
- drive-by правки соседнего кода/текста;
- работа без проверяемого критерия успеха.

Использовать как лёгкий gate для codebase, Pack, skill, protocol и document-cycle задач.

## Мини-протокол перед работой

Перед нетривиальными правками агент обязан назвать:
- `Допущения`: что считаю истинным и откуда это известно;
- `Критерий успеха`: что должно быть верно после работы;
- `Минимальный scope`: какие файлы/артефакты трогаю;
- `Не трогаю`: какие соседние хвосты, dirty files или домены вне задачи;
- `Проверка`: чем подтверждаю результат.

Для простой одноходовой задачи можно свернуть это в одну строку.

## Правила выполнения

1. **Не угадывать молча.**
   Если есть несколько трактовок, показать варианты или спросить. Если можно безопасно проверить локально, сначала проверить.

2. **Простота раньше гибкости.**
   Не добавлять абстракции, настройки, новые сущности и "на будущее", если задача решается существующим контуром.

3. **Хирургические изменения.**
   Каждая изменённая строка должна быть связана с задачей. Не форматировать и не улучшать соседний код/текст без отдельного scope.

4. **Убирать только свой мусор.**
   Удалять импорты, файлы и временные артефакты, созданные текущей работой. Не чистить чужие старые хвосты без команды.

5. **Проверяемый финал.**
   Завершать только после проверки: тест, diff, status, readback, визуальная проверка, Drive readback или другая проверка, подходящая домену.

## Goal-driven loop

Для задач длиннее одного очевидного действия агент работает коротким циклом:

1. **Plan**: сформулировать ближайший bounded step и проверку для него.
2. **Act**: выполнить только этот step.
3. **Verify**: проверить результат тем способом, который был назван заранее.
4. **Adjust**: если проверка показала расхождение, исправить в том же scope или остановиться и пересогласовать.

Не переходить к следующему step, если текущий не проверен или критерий успеха расплылся.

## Когда остановиться

Агент обязан остановиться и явно сказать пользователю, что scope изменился, если:
- задача стала требовать новый репозиторий, новый контур или новые права доступа;
- проверка показала риск поломать чужие dirty changes;
- обнаружена развилка, где оба варианта меняют результат для пользователя;
- для продолжения нужно хранить секрет, токен или персональные данные.

## Шаблон короткого gate

```markdown
Допущения:
Критерий успеха:
Scope:
Не трогаю:
Проверка:
Loop:
```

## Когда можно не разворачивать gate

- ответ без правок;
- очевидный lookup;
- микрофикс одной строки без риска соседнего scope.

Даже тогда агент обязан сохранять принцип: минимум изменений, явная проверка, без drive-by refactor.
